# PIDEA (Python)

- **DOM / IDE:** `connection`, `chat`, `ide_agent_message`, `selectors/`, `automation/`.
- **Workflow-Port (ohne Node):** Paket `workflow/` — Git-Branch anlegen, Branch-Namen-Templates (`${task.id}`, `{{task.title}}`, `{{timestamp}}`), Shell-Befehle im Workspace. Persistente IDE-Jobs nutzen das über **`scheduler_jobs.ide_workflow`** (siehe `scheduler_jobs_workflow`).
- **Prompts:** portierter Snapshot in `content-library/prompts/`; Scheduler: `use_pidea_task_management_phases` oder `phase_prompt_paths` (`content_library_prompts.py`).
- **Scheduler-Standard (`ide_agent`):** Wenn beim Anlegen **kein** `ide_workflow` mitgegeben wird, setzt `schedule_job_create` **`use_pidea_scheduler_pipeline: true`**. Ablauf: **neuer Chat** + `task-analyze.md` → **derselbe Chat** + `task-create.md` → **Git-Branch** → **neuer Chat** + `task-execute.md`: Task-Pläne vom Datenträger wie in PIDEA — fehlen `git_repo_path` / `project_path`, versucht der Worker **`project_path_cdp.detect_project_path_from_ide_sync`** (Port von PIDEA: CDP → ``page.evaluate`` / ``window.vscode.workspace``), danach Glob ``docs/09_roadmap/tasks/**/*.md`` (`task_plan_bundle.py`). Abwahl: `use_cdp_project_path: false`. Optional `attach_task_plans_to_execute`, `task_plan_glob`, `scheduler_pipeline_include_review`. Pipeline aus: `ide_workflow: { "use_pidea_scheduler_pipeline": false }`.
- **PIDEA-Workflow-JSON (1:1):** `workflows_data/task-workflows.json` (+ `task-create-workflows.json`, `analysis-workflows.json`) aus dem PIDEA-Backend; `ide_workflow.pidea_workflow_name` (z. B. `task-execution-workflow`, `standard-task-workflow`) führt die Steps aus (`pidea_workflow_json.py` / `pidea_workflow_executor.py`). Git-/IDE-Schritte portiert; Analyse/Test-/npm-Schritte wie im Node-Backend nur eingeschränkt.

**Produktion (`ide_workflow.pidea_workflow_name`):** Kurze Workflows bevorzugen, damit nicht dutzende Schritte nur „skip“ (Node-only) durchlaufen. Praktisch: **`task-execution-workflow`** (Git + IDE) oder **`task-review-workflow`** / **`task-check-state-workflow`** (nur IDE). **`standard-task-workflow`** und Workflows mit **`extends`** (z. B. `feature-workflow`, `testing-workflow`) sind lang; viele Steps nach dem IDE-Teil sind im Python-Port nicht ausführbar und werden übersprungen — für wiederkehrende Jobs eher ungeeignet.

Prinzip: LLM entscheidet Strategie; System-Code führt Git/Shell mit Guards aus — kein blindes `git push --force` aus dem Modell.
